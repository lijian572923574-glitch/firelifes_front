## 1. 页面概述

忘记密码页为已注册用户提供安全、便捷的密码重置通道。

## 2. 业务流程

```
登录页 → 点击「忘记密码」
  ↓
忘记密码页
  ↓
输入手机号 → 获取短信验证码
  ↓
输入验证码 + 新密码 + 确认密码
  ↓
提交 → 后端校验
  ├─ 成功 → 自动登录 → 跳转首页
  └─ 失败 → Toast 提示错误原因
```

> **产品决策**：重置密码后自动登录。
> 理由：用户已通过短信验证码完成身份证明（手机物理持有 = 最高验证因子），
> 再要求手动登录属于无意义操作摩擦。行业主流 App 均采用此模式。

## 3. 校验规则

### 3.1 前端校验
| 项 | 规则 |
|---|---|
| 手机号 | 11 位数字，格式合法 |
| 验证码 | 6 位数字，60s 内不可重复发送 |
| 新密码 | 6-20 位，字母+数字组合 |
| 确认密码 | 必须与新密码完全一致 |

### 3.2 后端校验
| 项 | 规则 |
|---|---|
| 手机号 | 必须已注册，未注册提示「该手机号未注册」 |
| 验证码 | 与数据库匹配、未过期、未使用 |
| 发送频率 | 同一手机号 60s 内不可重复发送 |
| 日上限 | 同一手机号每日发送上限 10 条 |

## 5. 交互规范

### 5.1 返回
- 左上角 ← 按钮 → `uni.navigateBack()`
- 容错：失败时 `redirectTo` 登录页

### 5.2 验证码发送
- 手机号格式校验通过才允许发送
- 发送成功后按钮变为 60s 倒计时
- 发送失败 Toast 提示原因

### 5.3 密码可见性
- 新密码输入框右侧  切换显示/隐藏
- 确认密码输入框不提供切换（默认不显示）

### 5.4 提交与自动登录
- 所有字段必填且校验通过才可提交
- 提交中按钮显示「重置中...」且不可点击
- 后端重置密码成功后直接返回 token + user
- 前端调用 `userStore.setAuth(token, user)` → `uni.reLaunch` 跳转首页
- 失败：Toast 提示原因

## 6. API 设计

### 6.1 发送验证码
```
POST /auth/send-sms-code
Body: { phone: string, type: 'reset-password' }
```

### 6.2 重置密码（成功后自动登录）

```
POST /auth/reset-password
Body: { phone: string, code: string, password: string }

Response: {
  token: string,    // JWT，后端签发
  user: {
    id: number,
    username: string,
    phone: string,
    nickname: string,
    avatarUrl: string
  }
}
```

> 与登录接口返回格式一致，前端直接 `setAuth(token, user)` + `reLaunch` 首页。

## 7. 技术要求
- 密码传输 HTTPS + 后端加盐哈希存储
- 重置成功后该验证码立即标记 used=true
- 禁止在 URL 中传递密码参数
- 重置成功后直接返回 JWT token，前端自动登录
