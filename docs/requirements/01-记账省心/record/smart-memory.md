## 功能概述

记账时自动记忆用户上次的选择，减少连续记账时的重复操作。包含两个维度：

1. **账户按分类记忆**：每个分类记住上次使用的账户，选分类时自动带出
2. **日期持续记忆**：记账成功后不重置日期，下一笔默认沿用上一笔的日期

### 为什么要做

用户补录整月支出时，每笔都要重新选择账户和日期，操作繁琐。行业主流做法（钱迹、秒速记账）均已支持类似功能。核心场景：

- 补录5月支出：用微信记了10笔餐饮，每笔都要选"微信"账户和"5月XX日"日期
- 连续记账：今天多笔支出，每笔都要确认日期是"今天"

### 行业参考

| App | 策略 |
|-----|------|
| 钱迹 | 每个分类记住上次使用的账户，选分类时自动带出 |
| 秒速记账 | 按分类记忆账户，同时根据时间段预测分类 |
| 随手记 | 模板记账，预填所有字段 |

## 用户故事

### 账户记忆
作为用户，当我选择了「餐饮」分类并使用「微信」账户记了一笔支出后，下次再选「餐饮」时，希望账户自动选为「微信」，而不是每次都回到默认账户。

### 日期记忆
作为用户，当我把日期改为5月15日记了一笔账后，下一笔记账时希望日期仍然是5月15日，而不是自动跳回今天。这样我补录同一日期的多笔支出时，不需要每次都改日期。

## 详细设计

### 一、账户按分类记忆

#### 核心逻辑

```
选择分类 → 查找该分类上次使用的账户 → 自动选中
          ↓（无记忆记录）
          回退到现有逻辑（isDefaultExpense / isDefaultIncome）
```

#### 记忆维度

| 场景 | 记忆Key | 记忆Value | 示例 |
|------|---------|-----------|------|
| 支出 - 选分类 | `expense_{categoryId}` | accountId | 餐饮→微信、交通→现金 |
| 收入 - 选分类 | `income_{categoryId}` | accountId | 工资→储蓄卡、红包→微信 |
| 转账 - 转出 | `transfer_from` | accountId | 转出→储蓄卡 |
| 转账 - 转入 | `transfer_to` | accountId | 转入→微信 |
| 还债 - 还款账户 | `repayment_from` | accountId | 还款→储蓄卡 |
| 还债 - 债权账户 | `repayment_to` | accountId | 债权→信用卡 |

#### 记忆时机

**写入**：记账成功后，将当前选择的账户写入对应Key
**读取**：选择分类时，先查记忆Key，有则用记忆值，无则回退现有默认逻辑

#### 兼容性

- 记忆的账户可能已被删除/隐藏，需校验：如果记忆的账户不在当前可用列表中，清除该条记忆，回退默认逻辑
- 新用户无记忆数据，行为与现在完全一致

#### 数据存储

使用 `uni.setStorageSync` / `uni.getStorageSync` 存储在本地，Key 前缀 `record_memory_`：

```
uni.setStorageSync('record_memory_expense_5', '3')     // 支出分类5 → 账户3
uni.setStorageSync('record_memory_income_12', '7')     // 收入分类12 → 账户7
uni.setStorageSync('record_memory_transfer_from', '1') // 转出 → 账户1
uni.setStorageSync('record_memory_transfer_to', '2')   // 转入 → 账户2
uni.setStorageSync('record_memory_repayment_from', '1') // 还款 → 账户1
uni.setStorageSync('record_memory_repayment_to', '5')   // 债权 → 账户5
```

### 二、日期持续记忆

#### 核心逻辑

```
当前行为：记账成功 → resetForm() → 日期重置为今天
期望行为：记账成功 → 日期保持不变 → 下一笔沿用上一笔日期
```

#### 规则

| 场景 | 日期默认值 |
|------|-----------|
| 首次进入记账页 | 今天 |
| 记账成功后（连续记账） | 沿用上一笔的日期 |
| 从其他页面重新进入记账页 | 今天（新一次记账会话开始） |
| 恢复草稿 | 草稿中保存的日期 |

#### 关键区分

"连续记账"和"重新进入"是两个不同的场景：

- **连续记账**：记完一笔不离开页面，继续记下一笔 → 日期不变
- **重新进入**：关闭记账页后再次打开 → 日期重置为今天

这样设计的原因：用户关闭再打开，通常意味着隔了一段时间，日期大概率是今天；而连续记账时，用户大概率在补录同一日期的账目。

#### 实现方式

记账成功后，不再调用 `resetForm()` 重置日期，而是：
1. 清空金额、备注
2. 保留分类、账户、日期
3. 光标聚焦金额输入框，方便直接输入下一笔

当页面通过 `onShow` 重新显示时（非记账成功触发的），才重置日期为今天。

## 交互流程

### 连续记账流程（改动后）

```
用户点击记账 → 进入记账页（日期=今天）
  → 选分类「餐饮」→ 账户自动带出「微信」（记忆）
  → 输入金额35 → 日期5月25日 → 提交
  → 记账成功
  → 金额清空、分类/账户/日期保持
  → 用户继续选分类「交通」→ 账户自动带出「现金」（记忆）
  → 输入金额5 → 日期仍是5月25日 → 提交
  → 记账成功
  → ...
```

### 补录历史账目流程

```
用户点击记账 → 进入记账页（日期=今天）
  → 改日期为5月15日
  → 选分类「餐饮」→ 账户自动带出「微信」（记忆）
  → 输入金额28 → 提交
  → 记账成功
  → 日期保持5月15日
  → 选分类「购物」→ 账户自动带出「支付宝」（记忆）
  → 输入金额199 → 提交
  → 日期仍是5月15日，无需重复选择
```

## `record-memory.ts` 接口设计

```typescript
/**
 * 记账智能记忆工具
 * 本地存储分类→账户映射，减少连续记账时的重复操作
 */

// 写入：记账成功后调用
function saveAccountMemory(
  type: 'expense' | 'income' | 'transfer' | 'repayment',
  categoryId: number,
  accountId: string
): void

// 读取：选分类时调用，返回记忆的账户ID，无记录返回null
function getAccountMemory(
  type: 'expense' | 'income' | 'transfer' | 'repayment',
  categoryId: number
): string | null

// 清除某条记忆（记忆的账户已不可用时）
function clearAccountMemory(
  type: 'expense' | 'income' | 'transfer' | 'repayment',
  categoryId: number
): void
```

## `record/index.vue` 改动点

### 1. `selectCategory()` — 选分类时查记忆

```
现有逻辑：
  选分类 → getAccountList() → isDefaultExpense/isDefaultIncome 选默认账户

改动后：
  选分类 → getAccountList() → 先查记忆 → 有记忆且账户可用 → 选中记忆账户
                                         → 无记忆或账户不可用 → 回退现有默认逻辑
```

### 2. `handleComplete()` — 记账成功后写记忆

```
现有逻辑：
  记账成功 → draft.remove() → resetForm()

改动后：
  记账成功 → saveAccountMemory(...) → draft.remove() → 部分重置（清金额/备注，保留日期/分类/账户）
```

### 3. 新增 `partialReset()` — 部分重置

```typescript
// 记账成功后的重置：只清空需要重新输入的字段
const partialReset = () => {
  displayAmount.value = ''
  remark.value = ''
  assetData.value = null
  // 以下字段保持不变：
  // selectedDate — 沿用上一笔日期
  // selectedCategory — 保持当前分类，方便继续记同分类
  // selectedAccount / fromAccount / toAccount — 保持当前账户
}
```

### 4. `onShow()` — 区分进入来源

```
onShow 触发时：
  - 如果是记账成功后的页面跳转（reLaunch 回来）→ 不重置日期
  - 如果是用户主动打开记账页 → 日期重置为今天

实现：用 flag 标记是否刚完成记账
```

## 边界情况

1. **记忆的账户已删除/隐藏** — 清除该条记忆，回退默认逻辑
2. **记忆的账户不在当前过滤范围内**（如收入时记忆了负债账户）— 清除记忆，回退默认逻辑
3. **用户切换支出/收入tab** — 清空分类和账户选择（与现有行为一致），下次选分类时按新tab类型查记忆
4. **转账/还债的双账户** — 各自独立记忆转出/转入、还款/债权
5. **本地存储被清除** — 无记忆数据，行为与未升级前完全一致
6. **跨日期补录** — 用户改了日期记一笔后，日期保持为改后的日期，继续补录同日期无需再改
7. **草稿恢复** — 草稿中保存的日期优先于记忆逻辑
8. **新用户** — 无记忆数据，行为与现在完全一致

## 与现有功能的关系

| 现有功能 | 影响 |
|---------|------|
| 默认账户（isDefaultExpense/Income） | 作为记忆的回退方案，无记忆数据时仍使用默认账户 |
| 草稿自动保存 | 草稿保存/恢复逻辑不变，草稿恢复时优先用草稿数据 |
| 记账成功跳转明细页 | 跳转行为不变，reLaunch回来时日期保持 |
| 编辑记录 | 不受影响，编辑页有独立的日期/账户逻辑 |
