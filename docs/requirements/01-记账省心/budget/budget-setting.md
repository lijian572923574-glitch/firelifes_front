## 一、功能概述

### 1.1 设计理念

基于「双预算体系」设计，包含两大核心模块：
- **常规月度预算**：日常消费类支出，每月相对固定
- **专项准备金**：年度大额周期性支出，按月平滑计提

解决传统预算的痛点：
- ❌ 某些月份因大额支出（学费、保险、旅游）导致预算崩溃
- ❌ 简单按「年度支出÷12」无法反映真实消费节奏
- ❌ 超支后产生心理挫败感，放弃预算管理

### 1.2 预算层级结构

```
年度总预算（顶层规划）
├── A. 常规月度预算（日常消费）
│   ├── 年度分类预算（如：餐饮36000元/年）
│   │   └── 自动拆解为月度预算（3000元/月）
│   └── 支持手动调整单月预算
└── B. 专项准备金（大额周期性支出）
    ├── 准备金项目（如：学费6000元/年）
    │   ├── 年度总金额
    │   ├── 预计支出月份
    │   └── 每月自动计提金额
    └── 记账时可选择"从准备金支出"
```

### 1.3 预算模式选型

| 模式 | 说明 | 本项目采用 |
|------|------|------------|
| 分类预算 | 为每个支出分类设定月度上限 | ✅ 常规月度预算 |
| 总预算 | 只设月度总支出上限 | ✅ 总预算兜底 |
| 准备金 | 年度大额支出按月平滑计提 | ✅ 专项准备金 |

采用「混合预算 + 准备金」双轨制，兼顾日常精细控制与大额支出平滑。

### 1.4 用户故事

**常规预算**：作为用户，我希望为餐饮、交通、购物等重点分类设定月度预算，同时设一个总预算兜底，这样我既知道钱花在哪，又能整体控制支出。

**专项准备金**：作为用户，我希望把学费、保险费、旅游基金等年度大额支出按月拆分计提，这样每个月都不会因为一笔大额支出导致预算崩溃。

### 2.6 设计评审反馈与修正

> 以下问题在全员评审中发现并修正，v1.1 已采纳。

| # | 评审问题 | 修正方案 |
|---|---------|---------|
| 1 | **分类体系不匹配**：原用 `category_id: varchar(50)` 关联系统分类表 `categories.id`，但记账使用 `typeId: int` 关联 `user_category_customizations.id` | 统一为 `type_id: int`，关联 `user_category_customizations` |
| 2 | **MVP范围过大**：5张新表 + 准备金 + 定时任务 + 预警表，一次性交付风险高 | 分期交付（见第十二章） |
| 3 | **总预算 vs 分类预算关系不明确** | 总预算为**软参考**（提示用），分类预算为**硬限制**（预警用） |
| 4 | **预算复制策略不明确** | 增加 `is_base` 字段区分基础预算和临时调整；每月复制 `is_base=true` 的记录 |
| 5 | **budget_logs / budget_alerts 可延后** | v1.0 MVP 用实时聚合 + 前端预警；v2.0 再持久化 |

## 三、核心业务规则

### 3.1 常规预算业务规则

#### 3.1.1 预算创建
- 支持创建年度总预算、年度分类预算、月度分类预算
- 创建年度分类预算时，自动拆解为12个月的月度预算
- 支持手动调整某个月的预算（如：某月增加购物预算）
- 同一用户同一周期同一分类只能有一个生效预算
- 预算周期采用**自然月**（每月1日~月底）

#### 3.1.2 预算扣减逻辑（与记账联动）
```
每笔记账（支出类型）创建时：
1. 按记账日期匹配对应月份的预算
2. 按分类优先匹配（通过 record.typeId 匹配 budget.type_id）：
   ├── 匹配分类预算（如：餐饮分类预算，type_id = record.typeId）
   ├── 匹配分类组预算（如：美食组预算，category_group_id = 该分类所属group）
   └── 匹配总预算（type_id = NULL 兜底）
3. 依次扣减各级预算额度（事务内 UPDATE budget SET spent = spent + ?）
4. 计算使用百分比
5. 前端根据百分比判断颜色和预警提示（v1.0 不持久化预警）
6. 记账接口返回 budgetImpact（受影响预算、当前使用百分比）
```
> **说明**：v1.0 MVP 阶段，budget_logs 和 budget_alerts 不持久化。预算扣减只更新 `budget.spent`，使用明细通过实时聚合 record 表获取。预警由前端根据 `spent/amount` 百分比判断。

#### 3.1.3 预算调整
- 月中可随时调整预算金额
- 调整后进度条实时重新计算
- 调整记录写入日志

#### 3.1.4 预算复制
- 每月1日检查当月是否有预算记录
- 无 → 自动复制上月**基础预算**（`is_base = true` 的预算记录），金额不变，`spent` 归零，`is_base = true`
- 有 → 直接使用
- **关键区分**：用户月中临时调高的预算（如春节购物从¥1,500调到¥3,000），`is_base = false`，不会被复制到下月。下月仍复制基准预算 ¥1,500

#### 3.1.5 总预算与分类预算的关系
- **总预算**（`type_id = NULL`）：软参考。用于"本月总计是否超标"的整体感知，不强制约束
- **分类预算**（`type_id` 有值）：硬限制。达到阈值时触发预警
- **未设预算的分类**：走总预算兜底展示，不触发独立预警
- 创建分类预算时，如果分类预算总和超过总预算，提示用户但不阻止保存

### 3.2 准备金业务规则

#### 3.2.1 准备金计提
- 每月1日自动计提所有生效准备金的月度金额
- 计提金额 = 年度目标金额 ÷ 12
- 支持手动调整某月计提金额（如：某月多计提）
- 支持一次性计提多个月金额

#### 3.2.2 准备金支出
```
记账时可选择"从准备金支出"：
1. 选择准备金项目
2. 支出金额从准备金余额中扣减
3. 不计入当月常规预算（避免双重扣减）
4. 准备金余额可为负（表示已透支使用）
5. 后续月份继续计提，自动补足负余额
```

#### 3.2.3 准备金调整
- 支持手动调整准备金余额（如：年终有结余）
- 调整时需记录调整原因
- 准备金结余可结转下一年或转入储蓄

### 3.3 预警触发规则

| 级别 | 阈值 | 颜色 | 说明 |
|------|------|------|------|
| 正常 | < 60% | 🟢 绿色 #00b894 | 预算健康 |
| 关注 | 60%~80% | 🔵 蓝色 #00BFFF | 正常消耗 |
| Level 1 | 80% | 🟡 橙色 #fdcb6e | "接近预算，请留意支出" |
| Level 2 | 95% | 🟠 深橙 | "即将超支，建议控制消费" |
| Level 3 | ≥100% | 🔴 红色 #e17055 | "已超支！预算已用完" |

- 每级别预警只触发一次（跨级别可再次触发）
- 支持用户自定义各预算的预警阈值

## 四、API 接口设计

> 所有接口统一返回 `{ "success": true/false, "message": "...", "data": {} }`，均需JWT认证。

### 4.1 常规预算接口 (/api/budgets)

#### 4.1.1 创建预算
```
POST /api/budgets
```
```json
{
  "budgetType": "normal",
  "name": "餐饮月度预算",
  "typeId": 1,
  "periodType": "monthly",
  "year": 2026, "month": 5,
  "amount": 3000.00,
  "alertThreshold": 80,
  "alertEnabled": true
}
```

#### 4.1.2 批量创建年度预算（自动拆解12个月）
```
POST /api/budgets/yearly/batch
```
```json
{
  "year": 2026,
  "budgets": [
    { "typeId": 1, "yearlyAmount": 36000.00, "alertThreshold": 80 },
    { "typeId": 2, "yearlyAmount": 9600.00, "alertThreshold": 80 }
  ]
}
```

#### 4.1.3 更新预算
```
PUT /api/budgets/:id
```
```json
{
  "name": "餐饮月度预算（调整）",
  "amount": 3500.00,
  "alertThreshold": 85,
  "isActive": true
}
```

#### 4.1.4 删除预算
```
DELETE /api/budgets/:id
```

#### 4.1.5 获取预算列表（按月份）
```
GET /api/budgets?year=2026&month=5&budgetType=normal
```
返回：当月总预算、总已用、总剩余、各分类预算明细（含 alertStatus）

#### 4.1.6 获取预算详情（含使用明细）
```
GET /api/budgets/:id
```
返回：预算详情 + usageLogs（每次扣减记录）

#### 4.1.7 获取当前预算概览
```
GET /api/budgets/overview/current
```
返回：当月预算总览、日均可用、预计月末余额、有预警的预算列表

### 4.2 准备金接口 (/api/reserve-funds)

#### 4.2.1 创建准备金项目
```
POST /api/reserve-funds
```
```json
{
  "name": "孩子学费", "year": 2026,
  "targetAmount": 6000.00, "expectedMonth": 9,
  "typeId": 3, "description": "每年9月交学费"
}
```
> 系统自动计算 `monthlyAccrual = targetAmount ÷ 12`

#### 4.2.2 更新 / 4.2.3 删除准备金
```
PUT /api/reserve-funds/:id    DELETE /api/reserve-funds/:id
```

#### 4.2.4 获取准备金列表
```
GET /api/reserve-funds?year=2026
```
返回：总目标金额、总当前余额、各准备金明细（含累积百分比、剩余月份）

#### 4.2.5 准备金详情（含计提记录）
```
GET /api/reserve-funds/:id
```

#### 4.2.6 手动计提
```
POST /api/reserve-funds/:id/accrue
{ "amount": 500.00, "year": 2026, "month": 5, "remark": "5月计提" }
```

#### 4.2.7 批量计提当月（定时任务）
```
POST /api/reserve-funds/batch/accrue-current-month
```

#### 4.2.8 调整准备金余额
```
POST /api/reserve-funds/:id/adjust
{ "adjustAmount": 100.00, "remark": "年终结余转入" }
```

### 4.3 预算预警接口 (/api/budget-alerts) — ⏳ v2.0 实现

> v1.0 MVP 阶段，预警由前端根据 `spent/amount` 百分比实时判断，不持久化预警表。<br>
> v2.0 实现以下接口持久化预警。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/budget-alerts/unread | 获取未读预警列表 |
| PUT | /api/budget-alerts/:id/read | 标记已读 |
| PUT | /api/budget-alerts/read-all | 全部已读 |
| GET | /api/budget-alerts/history | 预警历史（分页） |

### 4.4 记账集成接口

#### 4.4.1 记账时预算联动（POST /record 扩展响应）
```json
{
  "recordId": 1001,
  "budgetImpact": {
    "affectedBudgets": [
      { "budgetId": 1, "name": "餐饮", "spent": 2035.00, "amount": 3000.00, "usedPercentage": 67.83 }
    ],
    "warnings": [
      { "budgetId": 2, "name": "购物", "level": "warning", "message": "购物预算已使用85%", "usedPercentage": 85 }
    ]
  }
}
```
> v1.0: `warnings` 由前端根据 `usedPercentage` 判断；v2.0: 由后端持久化预警后返回

#### 4.4.2 记账时选择从准备金支出 — ⏳ v1.5 实现
```json
{
  "type": "expense", "amount": 6000.00,
  "typeId": 3, "date": "2026-09-01",
  "useReserveFund": { "reserveFundId": 1, "amount": 6000.00 }
}
```

### 5.4 交互流程

```
1. 首次进入预算模块
   └── 无预算数据 → 引导创建：
       ├── 输入月收入 → 自动建议总预算（月收入×70%）
       ├── 从常用分类中选择需要预算的分类
       └── 一键创建预算

2. 编辑预算
   ├── 修改总预算金额 → 实时校验：分类预算总和 ≤ 总预算（可配置）
   ├── 修改分类预算 → 实时更新
   └── 添加新分类预算 → 从支出分类列表选择

3. 准备金管理
   ├── 添加准备金 → 输入年度目标 → 自动计算月计提额
   ├── 编辑准备金 → 修改目标金额 → 重新计算月计提额
   └── 删除准备金 → 确认 → 余额处理（转回净资产或放弃）
```

## 十一、边界情况处理

| 场景 | 处理方案 |
|------|----------|
| 某月忘记计提准备金（v1.5） | 支持手动补提 |
| 准备金余额为负（v1.5） | 后续月份计提自动补足 |
| 分类预算总和超过总预算 | 提示用户，但不阻止保存（总预算为软参考） |
| 记账记录删除/修改 | 事务内回溯调整对应 budget.spent |
| 年中才开始使用预算 | 支持从任意月份开始创建当月预算 |
| 准备金年度有结余（v1.5） | 支持结转下年或转入储蓄 |
| 预算关联的分类被禁用/删除 | 关联预算标记失效，提示用户重新配置 |
| 跨月记账的预算归属 | 按记账日期归属对应月份，补记上月支出扣上月预算 |
| 随机预算复制（同一预算多次执行定时任务） | UNIQUE KEY (user_id, type_id, year, month, period_type) 防重 |
| 多设备同时记账 | spent 使用数据库行级锁或事务内 SELECT ... FOR UPDATE |

### 测试要点

| 场景 | 预期行为 |
|------|----------|
| 记账后预算 spent 正确累加 | spent = 原值 + 记账金额 |
| 删除记账后 spent 正确回退 | spent = 原值 - 删除记录金额 |
| 修改记账金额后 spent 差量调整 | spent = 原值 - 旧金额 + 新金额 |
| 修改记账分类后正确转移 | 旧分类 spent--，新分类 spent++ |
| 定时任务重复执行 | 不产生重复预算记录 |
| 并发记账 | spent 累计正确，无丢失 |
